## 一、授权模式

[OAuth 2.0](https://oauth.net/2/)是目前最流行的授权框架，允许用户授权第三方移动应用访问他们存储在另外的服务提供者上的信息（比如照片、视频、用户信息等），而不需要将用户名和密码提供给第三方移动应用或分享他们数据的所有内容。

Oauth2有四种授权方式：

*   授权码模式：第三方应用通过客户端进行登录，如果通过github账号进行登录，那么第三方应用会跳转到github的资源服务器地址，携带了client_id、redirect_uri、授权类型(code模式)和state(防止csrf攻击的token,可以不填)。随后资源服务器会重定向到第三方应用url并携带code和state参数，随后第三方应用携带code、client_id和client_secret再去请求授权服务器，先验证code是否有效，有效则发放认证token，携带该token可以取资源服务器上的资源。

*   简化模式：第三方应用通过客户端进行登录，通过github账号访问资源服务器，认证完成后重定向到redirect_uri并携带token，省略了通过授权码再去获取token的过程。

*   密码模式：直接向第三方应用提供资源服务器的账号密码，第三方应用通过账号密码请求获取资源服务器上的资源。会向第三方应用暴露账号密码，除非特别信任该应用。

*   客户端模式：不通过资源所有者，直接以第三方应用的秘钥和id获取资源服务器的token。

在我们框架中使用密码模式和授权码模式，自有的App通过密码模式进行登录，第三方App通过授权码模式进行登录

## 二、具体实现

### 2.1权限管理

权限管理采用以角色为基础的权限管理设计（即RBAC模型），对系统操作的各种权限不是直接授予具体的用户，而是在用户集合与权限集合之间建立一个角色集合。每一种角色对应一组相应的权限。一旦用户被分配了适当的角色后，该用户就拥有此角色的所有操作权限。这样做的好处是，不必在每次创建用户时都进行分配权限的操作，只要分配用户相应的角色即可，而且角色的权限变更比用户的权限变更要少得多，这样将简化用户的权限管理，减少系统的开销。

![1595905405997.png](https://upload-images.jianshu.io/upload_images/21580557-0d49aaa30354c5fd.png?imageMogr2/auto-orient/strip%7CimageView2/2/w/1240)


### 2.2客户端信息管理

将需要认证的客户端信息持久化到数据库中，为每个客户端设置账号密码、权限信息、支持的认证方式等信息。在请求令牌时会对客户端的信息进行校验，客户端信息和用户信息认证通过后返回令牌。

### 2.3授权码管理

第三方登录采用授权码模式，用户请求授权码，认证中心生成授权码，存储到数据库并返回给客户端。用户携带授权码请求令牌，从数据库中比对授权码，比对成功则销毁授权码并返回令牌。

### 2.4令牌管理

Oauth2原生的token是一串随机的hash字符串，存在两个缺点：

*   token验证需要远程调用认证服务器，效率低；

*   token无法携带用户数据。

因此采用JWT(Json Web Token)作为token使用，JWT分为三部分：

*   头部Header；

*   载荷Payload；

*   签名Signature。

这三部分均用base64Url进行编码。认证服务器通过对称或非对称的加密方式利用payload生成signature，并在header中申明签名方式。这样jwt可以实现分布式的token验证功能，即资源服务器通过事先维护好的对称或者非对称密钥（非对称的话就是认证服务器提供的公钥），直接在本地验证token，这种去中心化的验证机制非常适合分布式架构。jwt相对于传统的token来说，解决以下两个痛点：  1、通过验证签名，对于token的验证可以直接在资源服务器本地完成，不需要连接认证服务器；  2、在payload中可以包含用户相关信息，这样就轻松实现了token和用户信息的绑定；  如果认证服务器颁发的是jwt格式的token，那么资源服务器就可以直接自己验证token的有效性并绑定用户，这无疑大大提升了处理效率且减少了单点隐患。

## 三、具体方案

### 3.1 方案一

#### 3.1.1 流程图

![1595573114297.png](https://upload-images.jianshu.io/upload_images/21580557-f6b3e5ffd1d9a098.png?imageMogr2/auto-orient/strip%7CimageView2/2/w/1240)

#### 3.1.2 流程概述

1.  App进入客户端登录页面，采用密码模式向认证中心请求令牌，认证中心从用户中心查询用户信息，认证中心生成长令牌，将长令牌返回给客户端。

2.  客户端获得长令牌后携带长令牌访问后台资源

*   首先在网关层解析长令牌并对客户端权限进行校验，如果客户端无权访问该资源的权限，则拦截；

*   权限无误后网关对请求进行转发，转发到对应的资源服务器；

*   在资源服务器中解析长令牌获取用户权限，对请求访问的资源进行鉴权。

3.  权限认证通过则返回访问的资源

#### 3.1.3 优缺点

**优点：**

*   jwt基于json，是无状态的令牌，非常方便使用；

*   可以在令牌中自定义丰富的内容，易扩展；

*   通过非对称加密算法和数字签名技术，JWT防止篡改，安全性高；

*   资源服务使用JWT可不依赖认证服务器即可完成授权。

**缺点：**

*   服务端无法判断Token是否有效(退出登录、修改密码、修改角色权限、强制登出注销账号等)；

*   同时也无法判断刷新令牌是否有效；

*   重复登录策略(抢占式和协作式)。

#### 3.1.4 改进方案

1.  将 token 存入内存数据库：将每一个签发的JWT保存在REDIS上，当出现上述的需求时，就更新对应的JWT，如果客户端与REDIS中的不同，那么登录失败。这种方法虽然可以解决上述问题，但是此时JWT似乎变成了有状态的了，部分失去了JWT的优点。

2.  黑名单机制：使用内存数据库比如 redis 维护一个黑名单，如果想让某个 token 失效的话就直接将这个 token 加入到 黑名单 即可。然后，每次使用 token 进行请求的话都会先判断这个 token 是否存在于黑名单中。

3.  在签发、鉴别JWT中，不使用公共的秘钥，而是为每一个账户保存一个对应的秘钥，当客户端发来请求时，使用秘钥验证，如果失败说明这个JWT已经失效，可以实现上述的功能。然而，对于每一个请求都需要在数据库中查找对应的秘钥，这为服务器增加了负担。

4.  如果用户主动注销，或者主动修改密码，仅仅在客户端对之前的JWT删除。

### 3.2 方案二

#### 3.2.1 流程图

![1595572949259.png](https://upload-images.jianshu.io/upload_images/21580557-75dc9a2c87ad8a48.png?imageMogr2/auto-orient/strip%7CimageView2/2/w/1240)


#### 3.2.2 流程概述

1.  App进入客户端登录页面，采用密码模式向认证中心请求令牌，认证中心从用户中心查询用户信息，认证中心生成长令牌和短令牌，将长令牌存储到Redis中缓存，并将短令牌返回给客户端。

2.  客户端获得短令牌后携带短令牌访问后台资源

*   首先在网关层将短令牌置换为长令牌，并对客户端权限进行校验，如果客户端无访问该资源的权限，则拦截；

*   权限无误后网关对请求进行转发，转发到对应的资源服务器；

*   在资源服务器中解析长令牌获取用户权限，对请求访问的资源进行鉴权。

3.  权限认证通过则返回访问的资源。

### 3.2.3 优缺点

**优点：**

1.  将每一个签发的JWT保存在REDIS上，当出现用户注销等情况时，就更新对应的JWT，如果客户端与REDIS中的不同，那么登录失败，解决了方案一的问题。

2.  客户端存储短令牌，减小了客户端存储压力和传输压力，降低了信息泄露的风险；

3.  服务端缓存了用户的刷新令牌，可以后台完成令牌的刷新，减少客户端交互逻辑。

**缺点：**

1.需要在数据库中维护全部用户的令牌，JWT变成了有状态的，违背了设计的初衷；

2.生成两个令牌，每次请求都需要查询数据库。
